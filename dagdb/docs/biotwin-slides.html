Architecture under test

Three claims DagDB makes

1 ≤ 6-edge bound

Every node has at most 6 incoming edges. Forces hierarchical trees over fan-out stars.

2 Ranked DAG

Nodes layered by rank. Edges flow leaves → root, enforced by the daemon at CONNECT time.

3 Properties as shared nodes

A property value is a node, not a column. Many entities subscribe; one LUT flip drives population-level change.

Why this domain

The liver already is a 6-bounded ranked DAG

Hepatic microarchitecture isn't abstract. Cells touch at most 6 neighbours. Lobules tile into zones. Zones aggregate into functions. Functions feed into organ health.

The constraint DagDB imposes on graphs is the constraint biology already imposes on tissue.

The subscription pattern

One toxin node · 200 subscribers

In a classical property-bag graph DB, every hepatocyte would store its own has_APAP boolean — 600 writes to flip them all.

In DagDB, toxin_APAP is a single node. Zone 3 cells have an incoming edge from it. Flipping one LUT propagates to all 200 subscribers in the next tick.

Conclusion

What the liver experiment proves

  • A 6-bounded ranked DAG fits a real biological hierarchy without force-fitting.
  • Properties-as-nodes is a population-level write primitive — O(1) writes drive O(N) state changes.
  • Zone-specificity of damage comes from subscription topology. No per-cell code.
  • OR-gate zones give graceful degradation — 33 % cell loss doesn't kill the organ.
  • Evaluation at 10 M nodes: 18.6 ms per tick. Storage: 14.4 MB compressed.
My application is to create a bio-digital twin. Real physical systems have real structural constraints. That's the whole point.

What's next

Known holes · honest list

  • Grid ≥ 4096 (16.7 M slots) hangs at daemon init — Metal/shm sizing issue, root cause unknown.
  • Motif / subgraph match operators — the real architectural differentiator, not yet built.
  • Time-travel replay — codec exists, not wired as RECORD/REPLAY DSL verbs yet.
  • Property-bag ingestion from Neo4j-shaped input — no tooling.
  • LOAD validator is sequential Swift — 6.5 s at 10 M nodes. GPU-parallel version is next.

This is an amateur engineering project. Numbers from one laptop, no controlled benchmark, no peer review. The experiments are reproducible from the repo.
